Open Questions (Execution Layer)

12 alignment-meeting questions for the execution layer

These are the calls the PM, Conductor co-owners, and Platform PO need to make. The addendum does not pre-decide them.

1

Execution result storage: separate table (proposed) or denormalised on canonical record

§6 proposes separate. Trade-off: extra join cost vs. schema bloat on the canonical record. Eng + PO call.

2

Retry policy per task_type: uniform or category-specific

v1 has no retry. v2 needs per-type policy. Relationship-tier (v1.0 §3.4) and messaging-tier arguably never auto-retry; ops-internal and templated trip-prep can. Default mapping to confirm.

3

Event bus tech for on-event triggers

Postgres LISTEN/NOTIFY (proposed for v2), Redis Streams, or Kafka. Eng call, but throughput projections from product needed to pick the right tier.

Starting proposal: Postgres LISTEN/NOTIFY for v1/v2. The squad already runs Postgres, infra cost is zero, throughput is sufficient at expected v1 volume (under 100 events per minute). Graduate to Redis Streams or Kafka in v3 if throughput or fan-out demands it.

4

Executor registry storage progression

Hardcoded (v1), config table (v2), admin UI (v3). The progression is in §10; the calendar is product-paced. Does product want admin UI sooner?

5

Pause-resume model for long-running tasks (v3 scope)

In-memory state machine with periodic checkpointing, or explicit per-step persisted state. Affects which executors can run > 1 hour.

6

Cancellation cascading semantics

When a parent canonical task is cancelled, do all child executions stop, including in-flight API calls? §9 proposes yes for trip-state-changed = cancelled. Edge case: what about a partial-cancellation (one component of a trip is cancelled but the trip itself isn't)?

7

How execution failures route to humans

Three options: (a) same Review queue as suggestions (v1.0 §7.2), (b) a separate "blocked tasks" lane on the task board, (c) a per-owner inbox notification. §7 leans toward (b); confirm.

8

Sub-task spawn pattern

§8.3 (pre-arrival checklist) shows a composite executor spawning scheduled sub-tasks. The agent-subtask task_type from v1.0 Appendix F also produces sub-tasks. Should both use the same parent_task_id linkage (already on the v1.0 schema), or does composite-spawn need a distinct relationship type?

9

Trigger event payload schemas

Each on-event trigger needs a payload schema (what fields the executor reads from the event). v1 doesn't ship on-event so this is v2 prep work, but should the schemas live in this spec or in a separate event-bus spec?

10

Min acceptable result threshold per executor type

§9 mentions "v2 introduces a per-executor min acceptable result check". Is this a product concept (per task_type configurable) or an executor-internal concept (each executor implements its own bar)?

11

Operator-facing executor visibility

Does the operator see "running hotel_search_executor v1" on the task surface, or just "in progress"? Trade-off: transparency vs. UI noise. Recommendation: show on a debug panel, hide on the default task surface.

12

Composite executor failure-mode visibility

When a composite (AI proposes, human approves, AI sends) fails at the AI-send step after the human approved, what does the human see? Default proposal: a "we couldn't send your approved message" notification with the error payload visible.
