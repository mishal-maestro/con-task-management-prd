Triggers

When approved tasks actually fire

Every canonical task has exactly one trigger type, derived from the when field on the v1.0 schema. The field's four values map directly to four trigger types.

Four trigger types

Immediate

when value: immediately

Fires the instant approval_state flips to approved. The approval gate dispatches the executor synchronously when execution time is bounded (under a few seconds), or asynchronously with a callback for executors that run longer.

Backed by: Synchronous executor invocation, or async with callback for long-running

Scheduled

when value: scheduled:<ts>

Fires at the named timestamp. A scheduler worker polls the tasks_due_to_fire queue once per minute. Tasks whose scheduled_at has passed are dispatched to their bound executor.

Backed by: Worker polling a tasks_due_to_fire queue every minute

On-event

when value: on-event:<event>

Fires when a named event matches. The named events for v1 are enumerated below; each maps to a stream in v1.0 §4. At task approval time, the system registers a listener with the event bus.

Backed by: Pub-sub event bus (recommendation: Postgres LISTEN/NOTIFY)

Manual-fire (deferred)

when value: deferred

Fires when a human explicitly invokes the task. Used when a task is created and approved, but execution is gated on a manual fire. The executor binding is still resolved at approval time. The task sits in pending execution status until an operator clicks "fire now".

Backed by: Operator action from the task surface

Each trigger has a payload schema, which the bound executor reads as part of its input. Payload schemas live in a separate engineering doc, they are not specified here.

Immediate trigger

Used for the dominant flow: human approves a suggestion, executor fires immediately. Backing: the approval gate dispatches the executor synchronously when execution time is bounded (under a few seconds), or asynchronously with a callback for executors that run longer.

Example from v1.0 Appendix D: D.1 Aman Tokyo hotel search. when is immediately, executor is hotel_search_executor, fires the moment the TA clicks approve.

Scheduled trigger

Used when a task should fire at a future time. Examples: "Send anniversary message July 1", "Pre-arrival checklist T minus 10 days", "Day-1 check-in T plus 24 hours after arrival".

Backing: a scheduler worker polls the tasks_due_to_fire queue once per minute. Tasks whose scheduled_at has passed are dispatched to their bound executor. The queue is a thin index on the task_execution table, populated at task approval time.

Drift

If the worker stalls or the system is down, scheduled tasks may fire late. The reconciler sweep (§7) catches late firings and surfaces them to ops if the lag exceeds a threshold (proposed: 15 minutes for time-sensitive task types, 6 hours for ambient ones; thresholds product-tunable per task_type).

On-event trigger

Used when a task should fire in response to a named event somewhere else in the system. The named events for v1 are enumerated in the table above; each maps to a stream in v1.0 §4.

What about §4 streams that aren't in the trigger registry yet?

v1.0 §4 lists nine streams. Six map to v1 triggers above. Three are deferred to v2 or later: Outbound advisor messages (used for AI follow-up reminders, v2), AI agent runs (internal to the agent layer), Client profile mutations (slow-moving, low priority for triggered tasks). The deferred set can join the registry incrementally without re-architecting the bus.

Engineering call (open question, §12)

Event bus tech choice. Recommendation for v1: Postgres LISTEN/NOTIFY. The squad already runs Postgres, infra cost is zero, throughput is sufficient at expected v1 volume (under 100 events per minute). Graduate to Redis Streams or Kafka in v3 if throughput or fan-out demands it.

Manual-fire trigger (deferred)

Used when a task is created and approved, but execution is gated on a manual fire. Examples: a TA pre-approves a series of vendor outreach messages but wants to send them only when a specific decision lands.

The executor binding is still resolved at approval time. The task sits in pending execution status until an operator clicks "fire now" on the task surface, at which point execution proceeds as if the trigger were immediate.
