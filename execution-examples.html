E.1 Aman Tokyo hotel search (extends v1.0 Appendix D.1)

The v1.0 canonical task is shown in Appendix D.1. Once the TA approves, the execution contract resolves as follows:

Execution contract

Executor binding hotel_search_executor v1 Registry match on (task_type = booking-action, subtype = property-search, action pattern = "Search hotels...")

E.3 Pre-arrival checklist (extends v1.0 Appendix D.3)

The v1.0 task in Appendix D.3 is auto-tier. The execution contract:

Execution contract

Contract element Value How resolved
Trigger type scheduled From when = scheduled:T-10d-from-departure
Executor binding pre_arrival_template_executor v1 Registry match on (task_type = trip-prep, subtype = pre-arrival-checklist)
Runtime context Ops associate on the trip; TA on the trip added as watcher executor_kind = composite. AI renders the templated comms and surfaces an operator checklist; ops associate handles transfer confirmations, restaurant briefs, weather notes.
Approval and execution flow auto for the templated renders; assist for any client-facing sends routing_tier = Auto at the v1.0 level. The composite executor splits internally: auto for the system-side sub-tasks (rendering, scheduling), assist for any sub-task that sends a client-facing message.
Closure rule auto-close on sub-task completion task_type = trip-prep closes when all spawned sub-tasks are done.

The parent task moves to in_progress waiting on the sub-tasks. Each scheduled send fires its own task-completed event when done, which the parent listens for via the task-completed on-event trigger. When the last sub-task completes, the parent auto-closes.

Total lifecycle from approval to close: 48 days, three states, one execution attempt.
