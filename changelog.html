v1.0 (2026-05-13) — applying call feedback from the v0.2 PM review

Changes from v0.2, applied after the post-v0.2 review call with the Conductor PM. v1.0 is alignment-ready for the wider review with TA leads, Platform PO, and Conductor co-owners.

  • Exec Executive summary added at top of doc. Narrative form (Amazon-style), about one page, covers problem statement, 4-stage flow, five origin buckets, routing tiers, task-type-driven rendering, open questions, and what-this-doc-is-not. Designed to make the alignment review decidable in 2 minutes if the reader doesn't want to dig deeper.
  • v1/v2 gate (§7.1 Stage 3) Approval gate phased evolution rewritten. v1 (launch): every suggestion goes through human review; confidence_score captured from day 1 for telemetry / calibration only. v2 (post-calibration): auto-approve above a threshold (e.g. ~90%) per task type. Avoids the v1 failure mode the ops team called out previously (AI-spammed tasks they didn't ask for).
  • Surface flip (§7.2) Surface model flipped — in-context approval is now the primary flow (inline Claude-style "yes, yes, yes" approval within messaging, trip pages, agent runs, and member pages); the Review queue is the catch-up surface for what wasn't acted on in context. v0.2 had the Review queue framed as the primary surface.
  • +4 fields (§7.2) Four new canonical task fields: action (verb-phrase), actor (who or what executes), when (execution timing), and confidence_score (AI certainty 0-1, drives v1 telemetry + v2 auto-approval gating). Schema grows 19 → 23 fields. Appendix D worked records updated with the new fields populated.
  • task_type (§7.2) task_family renamed to task_type and elevated from "to be defined in the PRD" placeholder to a load-bearing concept. Type drives (a) where the task surfaces in the UI and (b) which fields are visible per surface.
  • Appendix F (new) Task types and rollout phases. Nine task_type categories phased into v1 / v2 / v3: v1 covers messaging, booking-action, trip-prep, and disruption-response; v2 covers member-context, ops-internal, and agent-subtask; v3 covers compliance-safety and discovery. Each row has example subtypes + where it surfaces + origin-bucket cross-references. Plus a Surface rollout summary aggregating surfaces per phase.
  • AI replies (§7.1) AI-suggested replies are tasks — explicit callout added. AI-proposed message replies travel the same 4-stage flow with one-click inline approval; canonical task created and auto-closed in the same step. Audit trail (who approved sending which message) preserved in one canonical record.
  • §8 trim (18 → 15) Merged three pairs of overlapping questions: VVIP bypass + VVIP detection (was q9 + q17) became one VVIP policy question; closure-confidence model + per-origin closure mapping (was q5 + q13) became one closure model question; soft-closed audit state + reopen loop (was q10 + q14) became one closure audit + recovery question.
  • q6 reframe (§8 q6) Was "Default views: ops associate vs TA"; now "Default filters and sort orders per role" anchored to owner + watchers ownership model. Removes the implication that we need separate per-role dashboard products.
  • §6.5 compress (routing) PR-review vs Incident analogy compressed from 5 paragraphs to 4 sentences; depth pushed to Appendix E walkthroughs.
  • E.2 fix (Appendix E.2) Weather walkthrough Stage 3 now shows v1 (pending review at incident-triage speed) alongside v2 (auto-approve above threshold). Previously read "auto-approval... no human gate at creation," which contradicted §7.1.
  • Reading guide (top of doc) Added a Reading guide block: exec summary (2 min); §3 + §7 for depth; §8 for decisions; appendices for reference.
  • Companion v1.0 (this site) Companion app rebuilt from v0.2 to v1.0 in parallel: vocabulary + schema brought in line, BPMN updated for the 4-stage flow with v1/v2 phased approval, new Appendix F page, exec summary + reading guide on Overview. Notion v1.0 page remains the primary review surface.

v0.2 (2026-05-12) — applying PM feedback from v0.1 review

Changes from v0.1, applied row-by-row against the PM's emailed feedback. Each entry references the row ID from the feedback inventory.

  • K3 (§6 intro) — Clarified that auto / assist / escalate describe how a task is executed, distinct from how it came into existence (the creation approval gate covered in §7.1). Same three tiers, sharper framing.
  • C9 (§2) — Replaced Trip context with Primary context anchor. Broader concept (trip / message-conversation / account-member), explicit that internal-ops can be anchor-less in the data model but global is not a user-facing filter option (avoids catch-all dumping).
  • C10 (new §7.4) — Added Explainability sub-section. AI-suggested / AI-auto-approved tasks surface a one-line rationale + one evidence link. Hard constraint: one line + one link, not paragraphs. Manual tasks: rationale optional. Field-level definitions (rationale, evidence_refs) on the canonical task come in C6.
  • M1 (new §10) — Added Success metrics section with six signals: suggestion acceptance rate, auto-approved task accuracy, false positive rate, time to first action, completion / closure rate, reduction in missed operational moments. Signals only — no targets in this outline.
  • N1 (new Appendix B) — Added Task inventory (working list). Enumerates the task types pulled from §3.1–§3.5 plus a B.6 catch-all for manual-created tasks. Each task type tagged with typical creator (Customer / System / Human) and default routing tier. Explicitly flagged as not exhaustive — alignment meeting adds + prunes.
  • C1 (§2 + §3 intro) — Added Creation surface definition paired with Origin (origin = why, creation surface = where/how). Appended pairing sentence to §3 intro to make the distinction explicit. §3.5 rename to "(system-sensed)" considered and deliberately omitted — the PM's "ops-driven mixes manual + system" critique doesn't match §3.5's actual content (which is purely system-sensed). Defensive rename can be added later if alignment-meeting reviewers make the same misread.
  • C2 (§2 + §4 + §5 + new Appendix C) — Renamed WRITE stream → Event-driven stream and READ stream → Schedule-driven stream (plain English for what they actually are: push vs pull on the data dimension). Updated §4 and §5 headings, table column labels, intro and closing paragraphs to match. Engineering terms (WRITE / READ) retained as parenthetical references in §2 definitions and at the end of Appendix C. The PM's proposed Reactive / Proactive rename was rejected — those labels describe task intent (Dimension B), which is orthogonal to data direction (Dimension A). Conflating them would have introduced a meaning the original split didn't carry. Added Appendix C "Stream direction vs task intent" to spell out the orthogonality with a 4-cell example table, plus a forward-pointer noting that reactive/proactive can become operational at PRD-time (SLA defaults, priority, dashboard sort), just not in this outline.
  • C8 (§7.2) — Added Ownership model bullet: single primary owner (accountable for closure) + N watchers (visibility / collaboration, no accountability). The PM's proposed multi-assignee model was rejected because shared ownership tends to create a "both see it = neither owns it" failure mode where accountability gets diffused. Watchers meet the underlying visibility need without diffusing responsibility.
  • C6 (§7.2 + new Appendix D) — Added Canonical task fields block to §7.2 (19 fields covering the PM's 7 + the C8/C9/C10/C1/§6 fields + standard system fields), explicitly framed as data-layer-only with a hard UI constraint of ≤ 2 fields on manual create (title + owner). All other fields inferred / auto-filled to preserve CC v1's 2-second task creation. New Appendix D adds three worked examples (Client-driven Assist / World-driven Escalate / Trip-lifecycle Auto) showing every field populated with the "How populated" column — demonstrates the operator never types 19 fields in practice.
  • C7 (§7.2 + §8) — Surfaced TWO MVP alternatives for where suggested tasks live: (a) the PM's proposed central Review queue + clean board, or (b) contextual suggestions inline on the trip / message / account they came from, with silent 24h expiry. Both keep the board clean; the difference is whether suggestions cluster centrally or stay scattered. Did not pre-lock — added as new §8 open question 11. §7.2 intro updated from "single dashboard" to "task board" to match.
  • C4 / Piece 1 (§2 definitions) — Added the 4-stage model vocabulary: Signal, Suggestion, Approval gate, Canonical task. Replaced Task-shaped intent (subsumed by Signal). Removed Trigger (overlapped Signal) and System signal (subsumed by Signal). Chose to keep Signal and Suggestion as separate concepts (not collapsed per the PM's "signal / suggestion" framing) because the distinction earns its keep when multiple signals correlate into one suggestion or signals get filtered out as non-task-worthy. Definition explicitly notes that Signal and Suggestion are often simultaneous for simple cases.
  • C4 / Piece 2 (§7.1) — Replaced the 3-mode creation model (Manual / Automatic / Hybrid) with the 4-stage flow: Stage 1 Signal → Stage 2 Suggestion → Stage 3 Approval gate → Stage 4 Canonical task. Manual creation explicitly bypasses Stages 1–3. Agent mode is called out as a hybrid (AI sub-tasks travel the 4-stage flow within the agent run; human sub-tasks emerge as direct Canonical tasks). Stage-by-stage description includes the rationale + evidence surfacing at the approval-gate stage (per C10 Explainability).
  • C4 / Piece 3 (new §6.5 + new Appendix E) — Added Mental-model-per-origin section: PR-review semantics for Client-driven + Relationship; Incident semantics for World-driven + Trip Lifecycle + Internal ops. Explicitly explains why both analogies are needed (forcing one onto the other's work-shape produces predictable failure modes — dead suggestions under PR-review for time-sensitive alerts; auto-resolution under Incident for tone-sensitive relationship work). New Appendix E adds three worked walkthroughs: PR-review for a Client-driven booking request, Incident for a World-driven typhoon alert, and Incident with multi-signal correlation (3 inputs → 1 task) — which demonstrates the correlation property that motivated keeping Signal and Suggestion as separate concepts in Piece 1.
  • C4 / Piece 4 (Appendix A swimlane) — Updated the swimlane diagram to reflect the 4-stage flow: subgraphs are now Signals → AI processing → Suggestions → Approval gate → Canonical task (previously: Triggers → AI capabilities → Ops/TA review → Tasks system). The Approval gate subgraph now explicitly shows the auto-approval vs human-review branch. Manual creation is depicted with a dotted bypass arrow direct to Canonical task. Agent mode hybrid behaviour called out in the reading notes. Section heading updated from "who creates a task?" to "how a task is created" to match the flow framing.
  • C3 (§3.1 + §8 q12) — Split §3.1 Client-driven into 3.1a Explicit and 3.1b Behavioural / implicit. 3.1a covers stated requests (booking, emergency, future seed); 3.1b covers inferred preferences (from message content) and app-behavioural signals (taps, searches, saves). Added explicit rule: behavioural signals feed the taste graph by default; a Suggestion fires only when behaviour + threshold + trip-context coincide. Outcome table shows four representative behaviours mapped to their default outcomes. New §8 question 12 surfaces the behavioural-signal-scope decision as a product call (needs mobile-product input — threshold values not locked in this outline).
  • N3 (§8 q1) — Appended the Conductor PM's starting proposal for an MVP cut to §8 question 1: two lanes — (a) explicit client / ops tasks from messaging + core workflows, (b) high-confidence proactive tasks from Trip lifecycle + Internal ops. Behavioural signals, relationship moments, and broader ambient sensing deferred. Framed as a starting proposal for the alignment meeting to react to, with explicit note that it can be refined in the PRD — not a lock at outline grade.
  • R8 missing-items batch (§8 questions 13–17) — Added five open questions surfaced during internal review that the PM didn't flag: q13 closure-tier mapping per origin bucket, q14 AI/human conflict resolution on closure (close → reopen loop), q15 priority assignment rule, q16 workload-balancing enforcement (auto-route vs surface only), q17 VVIP detection rule. All five are gaps in the current outline that affect implementability, surfaced as open questions for the alignment meeting rather than pre-decided.
