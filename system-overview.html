System Overview

The full task pipeline at a glance: origin → stream → flow → routing → closure

↓

How to read this: a task moves through five layers, top to bottom. Click any card to expand it and see 2-3 concrete examples + a link to the relevant section. The pipeline shows the happy path; cross-cutting paths (manual bypass, agent-mode hybrid) are called out at the bottom.

Layer 1

Origin

— why does this task exist?
Client-driven
+−

Client tells us — explicitly or implicitly — that they need or want something.

Examples

  • 3.1a Explicit: "We need 4 nights at the Aman Tokyo for our anniversary, last week of October — suite with city view." → search task on the trip
  • 3.1a Explicit: Client lands in Mumbai, pre-arranged driver isn't there → re-dispatch transport + apologetic gesture from TA
  • 3.1b Behavioural: Client says "the Lutetia was way too modern for us" → save preference: avoid contemporary properties
Read more in §3.1 →
World-driven
+−

Something outside our system happens that affects an active or upcoming trip.

Examples

  • Weather: typhoon warning issued for Seoul, client arriving in 48 hrs → alert client, monitor hourly, prepare reroute scenarios
  • Schedule cancellation: airline cancels client's outbound the day before departure → rebook on partner carrier, alert client, coordinate fresh ground transfers
  • Award space: first-class award appears on client's saved LAX-SYD August dates → book within minutes before it disappears, then notify
Read more in §3.2 →
Trip lifecycle
+−

The trip itself, as it moves through pipeline or calendar, generates work.

Examples

  • Pre-arrival (T-10): trip enters final pre-departure week → spawn checklist (re-confirm transfers, brief restaurants on dietary notes, send weather + packing notes)
  • Day-of-departure (T-4h): trip is hours from wheels-up → "have a wonderful trip" message with confirmations + 24/7 emergency contact
  • Pipeline stall: trip sits in Booking stage > 5 days with no component changes → flag to ops manager
Read more in §3.3 →
Relationship
+−

Pure relationship moments, sourced from the client profile, surfaced by date or pattern.

Examples

  • Birthday (T-7): client's birthday in a week → TA hand-crafts a card or gift, ideally tied to a property or preference they love
  • Returning client (3rd trip): client books their 3rd trip with us → trigger personalised welcome amenity at the destination property
  • Care-context check-in: client mentioned a parent's chemo on a past trip → 6 weeks later, gentle "thinking of you" message before any commercial outreach
Read more in §3.4 →
Internal ops
+−

Our own workflow signals — surfaced by self-monitoring rather than external events.

Examples

  • Coverage gap: TA going on 2-week leave → reassign their 8 active trips, brief covering TAs, set OOO auto-replies
  • SLA breach forming: VVIP client's message unanswered > 90 min → escalate to ops manager and assigned TA in parallel
  • Workload imbalance: TA A has 12 active trips, TA B has 5 → recommend the next incoming request to TA B

Layer 2

Stream type

— how does the system find out about it?
Event-driven (push)
+−

Events arriving in our system from outside, continuously. The system reacts to what just happened.

9 streams (§4)

  • Inbound client messages — app chat, WhatsApp, email, voice transcript (sec-min)
  • External signals captured — weather, flight tracking, vendor data feeds, advisory feeds (min-hours)
  • Trip pipeline state changes — pipeline lane move, component edit, dates change (min-hours)
  • Booking confirmation events, Payment events, AI agent runs, Client profile mutations, Activity / audit events
Full table + BPMN in §4 →
Schedule-driven (pull)
+−

Queries we run on our own data on a schedule. The system goes looking for situations that warrant a task.

11 queries (§5)

  • Trips departing in N days — daily, spawns pre-arrival checklist
  • Saved searches: award space appears — 15-min, spawns "book award" alert
  • SLA timers approaching breach — continuous, escalate / prioritise
  • Birthday / anniversary in N days, Visa / passport expiring, Trips stuck in same stage, Workload check, etc.

"The dog that didn't bark" failure modes — clients we forgot, trips that quietly stalled, deals we missed.

Layer 3

4-stage creation flow (§7.1)

— how does it become a task?
1. Signal
+−

Raw event or sensed condition. Not a task yet — a candidate.

Examples

  • Client message arrives 09:43: "We need 4 nights at the Aman Tokyo..." The message classifier picks it up.
  • Weather feed pushes alert at 11:02: typhoon warning for Seoul. The signal monitor receives it.
  • Daily scheduled query at 00:01: finds Anderson Tokyo trip departing in 10 days.
Read more in §7.1 Stage 1 →
2. Suggestion
+−

System's proposed task. Draft fields + confidence_score. Lives in Review queue, not on the board yet.

Examples

  • Single signal → suggestion: Classifier promotes the booking message to Suggestion: title "Search: Aman Tokyo, suite, anniversary last week of Oct" + rationale + evidence link.
  • Multi-signal correlation → one suggestion: 3 signals about the same hotel issue (client message + vendor email + client follow-up) collapse into one suggestion with multi-signal evidence.
  • Stale suggestions expire silently after 24h default (see §8 q11).
Read more in §7.1 Stage 2 →
3. Approval gate
+−

Decision point. v1: human review every time. v2: auto-approve above confidence threshold.

Examples

  • PR-review pattern: TA reviews suggestion the way they'd review a code PR — sees rationale + evidence in one glance, edits if needed (e.g. narrows the property list), approves.
  • Incident pattern: Escalate-tier weather signal — human still ACKs at the gate in v1, but the action is "acknowledge an alert" measured in seconds, not "review a draft."
  • Reject: ends the lifecycle. No canonical task created.
Worked walkthroughs in App. E →
4. Canonical task
+−

Approved task record (29 fields). Lives on the task board. task_type drives where it surfaces in the UI.

Examples (App. D)

  • D.1 Client booking: task_type: booking-action, owner = TA on the trip, status = in_progress, priority = Normal.
  • D.2 Weather: task_type: disruption-response, watchers auto-added (ops manager), priority auto-bumped to High.
  • D.3 Pre-arrival: task_type: trip-prep, owner = ops associate, due_at derived from trip departure.

Layer 4

Routing tier (§6)

— who actually does the work?
Auto
+−

AI handles end-to-end. No human gate after creation. Mechanical, structured, low cost-of-error.

Examples

  • Pre-arrival checklist (T-10): AI sends templated brief to restaurants on dietary notes, weather + packing notes to client.
  • Compliance reminder: AI detects passport expiring < 6mo → sends renewal reminder.
  • Internal-ops: shift handoff summary auto-generated for next shift.
Read more in §6.1 →
Assist
+−

AI proposes; human reviews + approves before execution. Tone matters, judgment matters.

Examples

  • Client booking request: AI executes property search, TA reviews results before sending to client.
  • Vendor change: AI detects suite category closed for renovation, drafts client notice with alternatives within preference profile, TA approves.
  • Trip-lifecycle client comms: Day-1 check-in message — AI drafts in TA voice, TA approves before send.
Read more in §6.1 →
Escalate
+−

Immediate human attention. Time-sensitive, high-stakes (VVIP / >$X / safety), or AI confidence below threshold.

Examples

  • Typhoon warning Seoul, client arrives in 48hr: alert ops manager + TA in parallel. Watchers auto-added, priority auto-bumped to High.
  • Mid-trip emergency: client's pre-arranged driver isn't there → re-dispatch transport + apologetic gesture from TA.
  • Birthday / anniversary outreach: human voice irreplaceable. Even though it's scheduled, the gesture itself is human-only.
Read more in §6.1 →

Layer 5

Closure (§7.3)

— how does it leave the system?
Manual completion
+−

A human marks it done. Always available, lowest trust bar.

When

  • TA marks done after sending the booking confirmation to the client.
  • Ops associate marks done after re-confirming all transfers in pre-arrival checklist.
AI auto-close
+−

AI fully executed the task. Knows it's done because it did it.

When

  • Pre-arrival templated message sent → AI marks task done in same step.
  • Auto-tier compliance reminder dispatched → done.
  • Status transitions to done per the §7.2 status enum.
AI-proposed close
+−

AI detects evidence of completion, surfaces a "close this?" prompt for one-click human ACK.

When

  • Vendor confirmation arrives in inbox → AI surfaces "Close this booking task?" to TA.
  • Conversation moved past the issue → AI proposes close on the related task.
Manual dismissal
+−

Human closes without completion, with dismissal_reason captured. Distinct from "completed."

When

  • Operator dismisses with reason: "duplicate of TSK-1234".
  • Operator dismisses: "client cancelled the trip — no longer relevant."
  • Status transitions to dismissed; dismissal_reason required.

Cross-cutting paths (don't follow the happy path)

Manual bypass (Layer 1 → Layer 4)
+−

"Ops spots something" creates a task directly. Skips Layers 2 + 3 (Stream type, 4-stage flow). First-class fast path, not a fallback.

Examples

  • TA hovers a message and clicks "create task" — skips Suggestion + Approval gate, lands directly as Canonical.
  • Keyboard shortcut c available even when suggestions are firing — TAs who need to create a task NOW must never be forced through the suggestion pipeline.
  • Manual-create UI surfaces ≤ 2 fields (title + owner); everything else inferred from context.
Read more in §7.1 →
Agent mode hybrid (1 prompt → many tasks)
+−

"Ops prompts agent" can spawn both AI sub-tasks (full 4-stage flow) and human sub-tasks (direct to Canonical). One agent run, multiple tasks.

Schema linkage

  • AI sub-tasks carry parent_task_id + agent_run_id.
  • Human sub-tasks carry agent_run_id only — they're peers of the agent run, not children of any specific task.
  • Example: TA prompts "research connecting rooms at the Aman Tokyo" → agent spawns AI search task + human follow-up call task.
Read more in §7.1 agent-mode →
