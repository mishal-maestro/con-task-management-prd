Execution Layer Overview

Post-approval execution model, v0.1 draft

The v1.0 outline covers the four-stage flow up to and including approval, plus the canonical task schema and closure shape. It defines the execution metadata fields (action, actor, when, routing_tier) but does not describe the execution engine. Once a task is approved, what fires? How does the verb-phrase in action bind to a concrete function, API, or AI capability? What happens on failure? What fields capture the execution output? This addendum is the answer.

Execution layer definitions

Trigger

The condition that causes an approved canonical task to start executing. Derived from the task's when field (v1.0 §7.2). Three concrete types in v1: immediate, scheduled, on-event. A fourth, deferred, defers firing entirely until manual invocation.

Action

The verb-phrase work being done, as captured in the v1.0 action field ("Search Aman Tokyo", "Send follow-up reply", "Re-quote hotel after vendor change"). Free-text at the data layer, bound to a concrete executor at approval time.

Executor

The concrete worker that runs an action. Can be an AI capability (a named classifier / watcher / agent skill), a human queue (the assigned owner's inbox), or a composite of both (AI proposes, human approves, AI fires). Every approved canonical task has exactly one executor bound to it.

Executor kind

One of ai-capability, human-queue, composite. Maps to the v1.0 §6 routing tier as follows: auto typically resolves to ai-capability; assist resolves to composite; escalate resolves to human-queue. Exceptions are possible per task subtype.

Executor registry

A versioned table mapping (task_type, subtype, action_pattern) to a concrete executor configuration. Product-tunable. Resolution happens at approval time, the binding is then immutable for the life of the canonical task.

Execution contract

The resolved (trigger, action, actor, executor, routing tier) tuple for a specific canonical task. Locked at approval. Edits before approval, not after.

Execution attempt

One run of an executor against a canonical task. A task may have multiple attempts (retry, manual rerun) but only one execution contract. Recorded in the task_execution table.

Execution result

The output of a successful execution attempt, stored as a JSON payload in the task_execution row. Examples: ranked list of hotel matches, message receipt id, schedule confirmation id.

Reconciler sweep

A periodic job (every 24 hours in v1) that scans for tasks stuck in in_progress with no execution row updated within the window, then flags them as orphans for ops review. Catches "executor fired but never reported back" cases.

The Execution Contract

Every canonical task carries an execution contract, resolved at approval time and immutable thereafter. The contract has five elements, each derived from a v1.0 field:

Trigger type

Resolved from v1.0 field: when

Whether the executor fires immediately, at a timestamp, on a named event, or only on manual invocation.

Executor binding

Resolved from: action + task_type + subtype

Which concrete executor (function / API / skill / human queue) runs the work. Resolved via the registry.

Runtime context

Resolved from v1.0 field: actor

Who or what carries the work. For ai-capability executors this names the capability; for human-queue executors this names the operator inbox; for composite executors this names the human reviewer plus the AI capability.

Approval and execution flow

Resolved from v1.0 field: routing_tier

Auto fires the executor without further human gate. Assist runs the executor up to a draft state, then waits for human approval before continuing. Escalate skips AI execution entirely and routes to a human queue with high priority.

Closure rule

Resolved from: task_type (via Appendix F mapping)

How the task transitions to done (or blocked) once the executor returns. Wired in §7.

Agent layer boundary

The agent layer (Claude Agent SDK skills, workflows, orchestrator) is a separate spec, owned by the platform and agent squad. This addendum does not specify or constrain how that layer is built.

The task layer's contract with the agent layer:

  • • An executor of executor_kind = ai-capability invokes a named capability in the agent layer.
  • • The capability is identified by a slug stored in executor_config.capability_id. The slug is opaque to the task layer.
  • • The capability conforms to a synchronous interface (returns within a few seconds with a result payload or error) or an asynchronous one (returns an acknowledgement quickly, calls back later with task_id and payload).
  • • The agent layer is free to back the capability with whatever it chooses: a Claude Agent SDK skill, a LangChain workflow, a raw API call, a chain of calls. The task layer does not care.

Notion automations

Trigger plus condition plus action shape. The framing Sunthar called out on the May 14 standup. Useful reference for the on-event trigger model and for product-tunable action recipes.

GitHub Actions

YAML workflows defining triggers (on:), conditions (if:), and steps (each running a named action or composite). Useful reference for the executor registry pattern, especially the versioned action references.

Zapier or Make

Visual trigger-action chains. Useful reference for the operator-facing surface in v3 if product wants a no-code path for adding executors.

Temporal or AWS Step Functions or Celery

Long-running orchestration with persisted state, retry policies, and timeout handling. Useful reference for v3 pause-resume work.

Airflow

DAG-based scheduled task orchestration with retries and observability. Useful reference for the scheduled-fire side and the reconciler sweep pattern.
