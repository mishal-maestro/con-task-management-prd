Executors

Action-to-executor binding and v1 executor set

The v1.0 action field is a free-text verb-phrase ("Search Aman Tokyo, suite, anniversary last week of Oct"). The execution layer needs a deterministic way to bind that string to a concrete executor. This section proposes the binding model and enumerates the v1 executor set.

The Executor Registry

A versioned table, product-tunable, with the following columns:

Column Type Notes
executor_id string (slug) Primary key. Stable identifier, e.g. hotel_search_executor.
executor_kind enum ai-capability / human-queue / composite.
task_type string (FK to v1.0 Appendix F categories) The task category this executor serves.
subtype string (optional FK) The specific subtype within the category. Wildcard allowed (matches any subtype).
action_pattern string (regex or keyword list) Pattern matched against the canonical action field. Wildcard allowed.
executor_config JSON Capability-specific config (API endpoint, model name, queue name, etc.).
version integer Monotonic per (task_type, subtype, action_pattern). Resolution picks highest version.
active boolean Allows retiring executors without deleting history.

Resolution at approval time: query the registry with the canonical task's (task_type, subtype, action) tuple. Pick the highest-version active row that matches. Stamp executor_id and executor_version onto the canonical record. If no match, fall back to human_routing_executor and flag the task as unbound for ops review (the registry has a gap).

Why a registry table instead of code-only mapping. Three reasons. One, product needs to add executors for new task types without an engineering deploy in the longer run (v2 and v3). Two, A/B testing two executor configurations against the same action pattern is cleaner with a version column. Three, the audit trail (which executor version ran which task) lives on the row, not buried in a code commit. v1 may ship with the registry as a hardcoded fixture and graduate to a config table in v2.

v1 executor set (minimum viable)

Five executors, sized to cover the v1 task type set in v1.0 Appendix F. Each executor specifies its kind, input contract (fields read from the canonical task), output contract (what gets written back), failure modes, and rough latency.

message_send_executor

Kind: ai-capability
Latency: Sub-second to a few seconds

Purpose: Sends a templated or AI-generated message via the bound channel.

Default for task types: messaging (subtype scheduled-send, follow-up-prompt); trip-prep (subtypes day-of-departure-message, day-1-check-in, post-trip-review).

Input: primary_context (the conversation or trip), action (the message intent), evidence_refs (sometimes the source signal, included as context for templated rendering).

Output: Message receipt id (channel-specific), sent_at timestamp.

Failure modes: Channel API down, recipient blocked, content rejected by safety filter. All produce a failed execution row with the error code.

message_compose_executor

Kind: composite (AI drafts, human approves, send happens via message_send_executor as a chained step)
Latency: A few seconds for drafting, then human-bound for review

Purpose: Drafts a message for human review. Used for assist-tier messaging.

Default for task types: messaging (subtype reply-draft).

Input: primary_context (the conversation), recent message history, evidence_refs (any prior similar conversations the AI should reference).

Output: Draft message text written to a draft field on the canonical task. Task status moves to awaiting-review (a sub-state of in_progress for assist-tier; not a new top-level status).

Chained step: When the human approves the draft, message_send_executor fires with the approved text.

Failure modes: Drafting model unavailable, context retrieval failure. Falls back to a blank draft with the rationale visible to the operator.

hotel_search_executor

Kind: ai-capability
Latency: Seconds to tens of seconds

Purpose: Runs a hotel property search against the booking engine, returns a ranked list of options.

Default for task types: booking-action (subtype property-search).

Input: Property name (from action), dates and traveller details (from primary_context.trip), client preferences (from member profile linked to trip).

Output: Ranked list of hotel options as a JSON payload, attached to the task_execution result. Surface in the trip page's hotel section per v1.0 Appendix F.

Failure modes: Booking engine timeout, no inventory matches, invalid date range. failed execution row with the engine's error code. For no matches, return an empty result with a flag so the executor doesn't retry.

pre_arrival_template_executor

Kind: composite
Latency: A few seconds

Purpose: Renders and schedules a templated pre-arrival communication sequence and surfaces an operator checklist (re-confirm transfers, brief restaurants, send weather and packing notes per v1.0 §3.3).

Default for task types: trip-prep (subtype pre-arrival-checklist).

Input: Trip itinerary, weather forecast for destination, restaurant booking list, client dietary notes.

Output: A set of scheduled sub-tasks (each one a message_send_executor task wired to the scheduled trigger). Plus an inline checklist on the trip page.

Failure modes: Itinerary incomplete, forecast unavailable. Surfaces as a blocked task with the missing pieces called out, the executor does not auto-retry.

human_routing_executor

Kind: human-queue
Latency: Human-bound

Purpose: Fallback for escalate-tier tasks and for any (task_type, subtype, action) tuple without a more specific executor in v1.

Default for task types: disruption-response (all subtypes in v1); booking-action (subtypes quote-request, vendor-rebook, payment-chase in v1; will get specific executors in v2 to v3); unbound tasks fall back here.

Input: The full canonical record. The human has all 31 fields visible.

Output: None at the executor level. The human's action is captured by v1.0's existing operator essentials (status change, manual close, etc.).

Failure modes: None at the executor level. Failure is operator-level (missed task, SLA breach), surfaced via v1.0's existing internal-ops signals.

Every v1 task type from v1.0 Appendix F resolves to at least one v1 executor. Of the 15 v1 subtypes, eight resolve to specific AI capabilities and seven resolve to the human-routing fallback for v1, with specific executors arriving in v2 or v3.

Phased rollout

v1 (launch)

  • • Executor set: the five executors above. Half of the v1 subtypes resolve to human_routing_executor until specific executors land in v2 to v3.
  • • Executor registry: hardcoded in code. No admin UI, no config table. Updates require an engineering deploy.

v2

  • • Executor set: add specific executors for booking-action subtypes (quote-request, vendor-rebook, payment-chase) and for the v2 task types from v1.0 Appendix F (member-context, ops-internal, agent-subtask).
  • • Executor registry: graduates from hardcoded to a config table. Product can add executors via a config edit. Still no admin UI; that's v3.

v3

  • • Executor set: v3 task types from v1.0 Appendix F (compliance-safety, discovery).
  • • Executor registry: admin UI for product to add and version executors without engineering involvement. A/B testing two executors against the same action pattern becomes a product workflow.
