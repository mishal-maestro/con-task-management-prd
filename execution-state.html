Execution State Machine

Schema additions, status transitions, edge cases

The v1.0 canonical task carries 29 fields covering identity, ownership, lifecycle, context, and audit. It does not carry execution result fields (output payload, success or failure indicator, attempt counter, error trace). This addendum proposes a separate task_execution table joined to the canonical task by foreign key.

Why a separate task_execution table

Multi-attempt support

Retries are expected (transient API failures, scheduled fires that miss their window, manual reruns). One canonical task can have many execution attempts. Denormalising the latest attempt onto the canonical record loses the history.

Schema discipline

v1.0 §7.2 deliberately constrained the canonical record at 29 fields with UI implications. Bolting execution_status, result_payload, error_payload, attempt_count, and three timestamps onto the same row pushes the schema past 35 fields and complicates v1.0's rendering rules.

Audit and query patterns

Ops queries often look like "show me every failed payment-chase execution last week", which is a join-then-filter on the execution table, not a scan over the canonical record.

The cost is one join for surfaces that need the latest execution alongside the canonical record. Acceptable given the surface model in v1.0 §7.2 (the task board can defer execution detail to a detail-view panel).

task_execution table schema

Column Type Notes
execution_id string (primary key) Per-attempt unique id.
task_id string (FK to canonical task) The task this attempt belongs to.
executor_id string (FK to executor registry) Locked at approval time, copied here for audit.
executor_version integer Locked at approval time.
attempt_number integer, default 1 Increments on retry.
execution_status enum queued / running / succeeded / failed / cancelled.
triggered_by enum + payload immediate / scheduled-fire / event:<name> / manual-fire. For event: rows, includes the event payload reference.
started_at timestamp When the executor began.
completed_at timestamp, nullable When the executor returned (success or failure). Null while queued or running.
result_payload JSON, nullable Output on success. Examples: ranked hotel list, message receipt id, schedule confirmation id.
error_payload JSON, nullable Error code, error message, executor-specific diagnostic info on failure.
next_retry_at timestamp, nullable For retry-eligible failures. Null for succeeded, cancelled, and failed-and-final.

Execution state machine

The canonical task's status field (v1.0 §7.2) reflects the highest-level state. The task_execution row reflects the per-attempt detail. The v1.0 activity log records the high-level transitions; the execution table records the per-attempt details.

Status transitions

Execution event Canonical task status after Notes
Task approved, executor queued pending → in_progress (or stays pending for deferred trigger) The instant a non-deferred trigger registers the executor.
Executor starts running in_progress (no change if already in_progress) The execution row moves from queued to running.
Executor returns success, closure rule is auto-close in_progress → done For auto-tier templated tasks per v1.0 §7.3 (e.g. pre-arrival checklist on confirmation arrival).
Executor returns success, closure rule is AI-proposed-close in_progress (stays); banner on the task surface offers "close this?" Human ACKs to flip to done. v1.0 §7.3.
Executor returns success, closure rule is human ACK only in_progress (stays) Relationship-tier and other human-voice critical work. v1.0 §7.3.
Executor returns failure, retries available in_progress (stays); execution row marks failed, queues next_retry_at The reconciler picks it up.
Executor returns failure, retries exhausted in_progress → blocked; watchers notified Operator decides whether to re-scope (dismiss with reason re-scope) or manually retry.
Executor cancelled (upstream cancellation, e.g. trip cancelled) in_progress → dismissed; dismissal_reason = cancelled-upstream v1.0 §7.2 requires a dismissal reason.
Operator manually closes during execution in_progress → done or dismissed per operator action Standard v1.0 §7.3 paths still apply. The execution row is marked cancelled with triggered_by = manual-close.

Retry policy in v1

v1 is conservative. No automatic retry for any executor. Failure surfaces as a blocked task. The reconciler sweep notifies watchers. The operator decides whether to retry manually or re-scope.

Rationale: retrying without thinking burns API budget, can double-send messages, and can produce silent duplicates on the booking side. The squad does not have the metrics in v1 to know which executors retry safely. v2 introduces exponential backoff per executor type once the failure data tells us which retries are safe.

Reconciler sweep

A periodic job, every 24 hours in v1, scans for two problem cases:

  • 1. Tasks in in_progress with no task_execution row updated in over 24 hours. Likely orphans (executor fired but never reported back). Flag for ops review.
  • 2. Tasks in pending with a scheduled trigger whose scheduled_at is more than 15 minutes past due. Likely scheduler-worker lag. Flag for engineering review.

Both surface on a dedicated reconciler view, accessible from the ops dashboard. The view is not the task board; it is a separate audit surface for cases the normal flow should be catching but isn't.

Edge cases and failure modes

Case Handling in v1
Trigger fires but executor is unavailable (service down) Mark execution failed with a SERVICE_UNAVAILABLE code. Canonical task moves to blocked. Reconciler picks up. v2 introduces retry-with-backoff.
Executor succeeds but acknowledgement never returns Long-running executor stalls. The reconciler sweep flags after 24 hours (long-running task ceiling in v1). Ops escalates to engineering.
Executor fires twice (duplicate event) Trigger fingerprint on the event row prevents duplicate dispatch. Implementation: a content-hash on the trigger payload; if the same hash fires twice within a dedup window, only the first reaches the executor.
Two vendors confirm at once (correlated triggers) Per v1.0 §3.2 and Appendix E.3, the suggestion stage already correlates multi-signal cases into one suggestion. By the time the canonical task fires, the correlation has happened upstream. No special handling needed at the execution layer.
Client cancels the trip mid-execution A trip-state-changed event fires with new_state = cancelled. The execution layer listens for this on every active task bound to the cancelled trip and dispatches a cancellation to each running executor. The execution row moves to cancelled with triggered_by = manual-cancel-from-trip-cancellation. Canonical task moves to dismissed with dismissal_reason = cancelled-upstream.
Operator dismisses the task during execution Same as cancellation, but triggered_by = operator-dismiss. The executor receives a cancellation signal and returns gracefully if it can. If it can't (e.g. a message has already been sent), the dismissal still happens but the execution row records cancellation_came_too_late with the sent-message receipt id, preserving audit.
Scheduled fire is missed (worker down for an hour) The reconciler sweep flags scheduled fires more than 15 minutes past due. The operator decides whether to fire late or skip. v1 does not auto-fire late.
Executor returns a partial result E.g. hotel search returns 1 of 3 expected options. v1 treats this as succeeded with the partial payload. The TA sees the partial result and decides whether to re-search. v2 introduces a per-executor "min acceptable result" check; below the bar, the row is marked succeeded-degraded and surfaces a soft alert.
Long-running task crosses 24 hours Currently flagged by reconciler. v1 has no executor-side checkpointing. True pause-resume is v3 scope.
Task is approved with a future-scheduled trigger; then the operator wants to edit the action Per §3, the execution contract is immutable post-approval. The operator dismisses the current task with reason re-scope and creates a new suggestion. The activity log preserves the lineage.
Sub-task fails inside a composite executor's run The composite's parent task moves to blocked with the failing sub-task referenced in the error_payload. Sibling sub-tasks may have already succeeded; their execution rows remain on the record. v1 does not auto-rollback siblings.
